# PetPlanet 后端一致性修复设计

## 目标

修复后端种子图片、`getPet` 404 语义和单条病历/回忆删除后的上传文件清理，使 API 模式与本地 `PetRepository` 行为一致，同时不误删公共种子资源或仍在使用的文件。

## 方案比较

1. **关联表补种 + 引用感知清理（采用）**：保留现有数据模型，为种子记录写入静态图片 URL；删除记录后，仅清理 `assets` 表中登记、位于上传目录且已无引用的文件。改动集中，兼容现有前端。
2. **移除后端种子图片**：让前后端种子都不带照片。实现最简单，但会降低首屏效果并破坏当前视觉基线。
3. **后台垃圾回收任务**：定期扫描上传目录清理孤儿文件。适合更大规模系统，但首版需要额外调度与可观测性，超出本次范围。

## 数据与行为

### 种子图片

- `medical-1` 关联 `/assets/memory-sunlit-nap.jpg`。
- 八条回忆按前端 `src/data/seed.ts` 的照片 URL 逐条写入 `memory_assets`。
- 静态 `/assets/...` URL 不写入 `assets` 表，因此永远不会进入上传文件删除流程。
- `seedIfEmpty` 的主表和关联表写入保持在同一事务中。

### `getPet` 语义

- HTTP repository 请求宠物详情时，404 返回 `undefined`。
- 其他非 2xx 响应继续抛出带后端消息的 `Error`。
- 其他 repository 方法的错误语义不变。

### 上传文件清理

- 删除病历或回忆前读取其关联 URL。
- 在数据库事务中删除主记录，然后检查 URL 是否仍被病历、回忆或宠物头像引用。
- 仅当 URL 在 `assets` 表登记且引用数为零时删除对应 `assets` 行，并在事务提交后删除 `data/uploads/` 内的物理文件。
- 文件路径必须从受信任的 `assets.url` 文件名解析，并验证解析后的绝对路径仍位于上传目录内。
- 文件已不存在不影响删除接口成功；其他文件系统错误记录警告但不恢复已提交的数据库删除。

## 测试

- 内存 SQLite 测试验证种子病历与八条回忆的图片关联。
- HTTP repository 测试验证 404 返回 `undefined`，500 仍抛出错误。
- 临时上传目录测试验证单条病历/回忆删除会清理无引用文件；共享引用存在时文件保留。
- 最终运行全部测试、类型检查和生产构建。

## 非目标

- 不修改前端页面或视觉。
- 不引入定时垃圾回收服务。
- 不改造既有上传 API 或 URL 格式。
