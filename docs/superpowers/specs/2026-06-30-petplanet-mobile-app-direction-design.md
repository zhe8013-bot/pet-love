# PetPlanet 移动端 App 转向设计

## 状态

已完成产品方向评审，等待最终规格审阅。本文只定义设计与演进边界，不授权立即实施。

## 背景

PetPlanet 已具备宠物档案、健康病历、生活消耗、体重记录、生活照片、多宠切换、回忆模块以及本地/API 两种数据模式。当前实现功能较完整，但信息架构和视觉重心仍偏桌面管理工具：多宠卡片平均铺开、数据面板占主导、移动端主要依赖响应式收窄。

本次转向不删除既有功能，也不重写现有项目。目标是把 PetPlanet 重组为移动优先的宠物陪伴 App，并保留桌面端作为响应式扩展。

## 产品定位

PetPlanet 是一个高级、温馨的宠物生活管理 App。用户每天首先完成当前宠物的照护，再顺手记录健康、消费、体重和生活照片；长期沉淀的数据最终形成成长回顾与空间回忆墙。

产品闭环为：

`查看今天 → 完成照护 → 快速记录 → 数据沉淀 → 月度回顾与生活回忆`

“每日陪伴”只决定首页重心，不缩减产品范围。

## 目标

- 保留现有宠物档案、健康、生活、体重、回忆和多宠能力。
- 以 390×844 移动视口为主要设计与验收尺寸。
- 将当前宠物作为全局上下文，切换后所有页面同步更新。
- 建立可持续的今日照护模型，而不是从其他记录临时拼接待办。
- 使用奶油生活感作为全局底座，在首页与回忆页克制加入 2.5D 空间层次。
- 第一阶段继续使用 React/Vite 构建移动优先 PWA，稳定后再通过 Capacitor 获取原生能力。
- 将真实 3D 限制在用户主动进入的回忆墙，并按需加载。

## 非目标

- 不迁移到 React Native。
- 不重写 Express/SQLite 后端或 `PetRepository` 数据边界。
- 不在第一阶段实现完整离线写入队列、图片离线同步或多端冲突解决。
- 不把所有记录合并为一个通用 Event 数据表。
- 不在普通健康、生活和档案页面使用真实 3D。

## 全局信息架构

移动端底部保留五个一级入口：

1. **首页**：当前宠物、今日照护、快速记录、月度摘要和回忆入口。
2. **健康**：病历、复诊、用药、搜索筛选、医疗照片及现有新增、详情和删除能力。
3. **生活**：消费、分类统计、月份切换、体重趋势及历史记录。
4. **回忆**：生活照片、成长时间线、2D 相册及未来 3D 照片墙。
5. **宠物**：多宠档案、头像、提醒、编辑、删除和当前宠物切换。

### App Shell

- 顶部显示当前宠物切换器和全局新增入口；在通知中心真正实现前不放置无行为的通知按钮。
- 底部固定五项导航，并适配安全区。
- 全局新增入口打开统一底部面板，提供病历、消费、体重、照片和照护记录。
- 统一面板只负责选择记录类型与承载表单；各领域继续使用独立模型和 Repository 方法。
- 各一级页面仍保留与上下文相关的新增入口。
- 宽屏下底部导航可转为轻量侧栏，但移动端结构是唯一产品基准。

## 首页设计

首页不再平均展示多只宠物，也不作为功能目录。

### 1. 当前宠物主场

- 只突出当前宠物的大照片、名字、陪伴天数和今日状态。
- 其他宠物仅通过顶部切换器进入。
- 视觉使用奶油底色、真实照片、轻微轨道或遮挡关系，建立陪伴感与空间层次。

### 2. 今日照护

- 按时间展示喂食、散步、用药、梳毛、复诊等照护事项。
- 支持完成、稍后、跳过和撤销。
- 完成后即时更新今日进度；保存失败时回滚并提供重试。
- 复诊等医疗提醒可作为只读派生提醒进入今日视图，但不会替代正式照护计划。

### 3. 快速记录

- “记录一下”打开底部面板。
- 固定提供病历、消费、体重、照片和照护记录，不隐藏任何既有能力。
- 常用记录类型可以排序，但不得改变数据模型或功能可发现性。

### 4. 月度 Bento

- 使用一个主单元和三个次单元组成非对称摘要区。
- 展示体重变化、消费、健康记录和生活照片数量。
- 首页只读取聚合结果；点击后进入对应完整页面。

### 5. 生活回忆入口

- 使用真实照片叠层和轻微 2.5D 纵深展示当月故事。
- 默认进入 2D 成长相册。
- 用户主动切换后才加载 RTF 3D 照片墙。

## 其他一级页面

### 健康

- 以可信、安静、清晰为优先。
- 保留搜索、状态与时间筛选、详情、照片、复诊、删除确认和异常状态。
- 使用时间线和轻分隔组织内容，不使用后台表格。

### 生活

- 保留按月切换、消费汇总、分类、消费列表、体重趋势和体重历史。
- 图表只表达必要趋势，避免密集仪表盘。
- 消费与体重继续保持独立记录类型。

### 回忆

- 2D 相册和 3D 照片墙读取同一份 `Memory` 数据。
- 2D 是默认、可访问且低性能成本的完整体验；3D 是增强模式，不是功能前提。
- 共享月份/季节时间轴和当前选中回忆。

### 宠物

- 保留新增、编辑、删除、头像上传、提醒、健康摘要、体重和照护统计。
- 删除宠物继续使用明确二次确认，并级联处理相关数据。
- 视觉像高级个人档案，不做多卡片后台列表。

## 视觉系统

整体采用“高级奶油生活感为底，2.5D 只用于品牌记忆点”。

- 页面背景：奶油白 `#FAF6EE`。
- 主文字：深梅紫或墨褐。
- 品牌色：蜜桃珊瑚。
- 辅助色：低饱和鼠尾草。
- 薰衣草与烛光金仅用于回忆、特殊状态和少量高光。
- 标题使用温暖人文宋体；正文、标签和数据使用现代无衬线。
- 主卡圆角 22–24px，普通模块 16–18px，底部面板顶部圆角 28px。
- 阴影使用暖色、宽扩散、低不透明度；禁用厚重黑影和大面积玻璃拟态。
- 不使用大量彩色标签、卡片套卡片或装饰性噪音。

### 页面性格

- 首页最有陪伴感，可使用小范围视差、宠物轨道与照片遮挡。
- 健康页最克制，以信息可信度为主。
- 生活页最重视数据扫读，但保持生活化 Bento 结构。
- 回忆页情绪与照片占比最高。
- 宠物页采用编辑式档案排版。

### 动效与可访问性

- 普通交互使用 180–240ms 的淡入、位移和状态过渡。
- 首页只允许小范围 2.5D 视差，不影响信息阅读和点击目标。
- 3D 只在回忆页主动加载。
- 遵循 `prefers-reduced-motion`，关闭视差和非必要过渡。
- 文字、按钮、底部导航和状态颜色满足移动端可读性与触控尺寸要求。

## 领域模型修正

现有首页待办主要由 `Pet.reminder`、复诊记录和月度体重提示临时组合，不能支撑可重复照护、历史完成状态或可靠提醒。今日照护必须成为正式领域能力。

### CarePlan

表示可重复或一次性的照护计划：

- `id`
- `petId`
- `title`
- `category`: `feeding | walk | medication | grooming | other`
- `schedule`: 一次性日期，或按日/按周的本地时间规则
- `notes`
- `isActive`

### CareOccurrence

表示某个计划在具体时间的一次执行：

- `id`
- `planId`
- `petId`
- `dueAt`
- `status`: `pending | done | snoozed | skipped`
- `completedAt`
- `snoozedUntil`

计划以用户本地墙上时间配置；生成的 occurrence 使用 ISO 时间保存。调度逻辑必须覆盖跨日、时区变化和夏令时边界。

Occurrence 按需生成当天及未来 7 天的滚动窗口，并以 `planId + dueAt` 保证幂等，避免重复任务和无限预生成。

### 派生数据

- 今日页面通过专用 selector/service 聚合 CareOccurrence、复诊提醒和月度维护提示。
- 聚合结果是读取模型，不写入通用 Event 表。
- 病历、消费、体重、回忆和照护继续拥有独立生命周期。

## 数据流与边界

`PetRepository → PetDataProvider/查询层 → 页面 selector → UI`

- `PetRepository` 继续隔离本地模式和 API 模式。
- `PetDataProvider` 继续维护宠物集合和当前宠物上下文。
- 新增 CarePlan/CareOccurrence Repository 方法，但不改变其他领域的职责。
- 首页只消费聚合读取结果，不直接复制领域计算。
- 图片上传继续使用现有 AssetKind 和上传流程；失败时保留表单输入。
- 体重更新仍由 Weight 领域维护 `Pet.currentWeight` 的一致性。

## 本地数据迁移

新增照护模型后，本地状态从 `petplanet:data:v1` 迁移到 `petplanet:data:v2`。

启动顺序：

1. 若 v2 存在，验证后读取。
2. 若只有 v1，复制全部宠物、病历、消费、体重和回忆数据。
3. 初始化空的 `carePlans` 和 `careOccurrences`。
4. 写入 v2，并保留 v1 作为一次发布周期内的只读回退数据。
5. 迁移必须幂等；失败时不得覆盖原 v1 数据。

服务器数据库需要对应 schema 迁移，但不在 App Shell 重构中顺带改写其他表。

## PWA 与原生边界

### 第一阶段

- Web App Manifest、安装入口和应用图标。
- 缓存应用壳与必要静态资源。
- 离线时允许打开 App，并读取最后一次成功保存的数据。
- 显示明确离线状态。
- 不承诺离线新增队列、图片离线上传或冲突合并。

### 后续阶段

- 评估离线写入队列与同步协议。
- 稳定后通过 Capacitor 接入推送、相机、相册权限和应用商店分发。
- 原生封装不得复制业务逻辑；Web/PWA 仍是主要代码基础。

## 3D 回忆墙边界

- `MemoryPage` 提供完整 2D 模式。
- 3D 模块通过路由或模式切换动态导入。
- RTF/Three 依赖不得进入首屏主包。
- WebGL 不可用、性能不足或 reduced-motion 开启时提供 2D 降级。
- 2D 与 3D 使用同一 Memory 数据、时间轴和选中状态。

## 状态与错误处理

- 页面加载使用与版式匹配的骨架，不使用整页空白 spinner。
- 请求失败显示原因和重试入口，不静默回退到旧数据。
- 照护完成采用乐观更新，失败后回滚。
- 表单提交期间禁用重复提交；失败时保留输入。
- 删除宠物、病历、消费、体重和回忆继续使用统一确认流程。
- 区分无宠物、无记录、筛选无结果和离线四类状态。

## 实施阶段

1. **基线冻结**：确认现有测试、构建和关键流程通过，保存移动/桌面视觉基线。
2. **照护领域与迁移**：增加 CarePlan、CareOccurrence、Repository 能力和 v1→v2 迁移。
3. **App Shell**：移动顶部栏、五项底部导航、宠物切换和统一快速记录面板。
4. **首页重构**：当前宠物主场、今日照护、月度 Bento 和回忆入口。
5. **既有页面移动化**：依次调整健康、生活和宠物档案，不删除业务能力。
6. **回忆体系**：完成 2D 相册，再恢复并优化按需加载的 RTF 3D 增强模式。
7. **PWA**：安装能力、离线应用壳、缓存策略和明确离线状态。
8. **原生评估**：在 Web/PWA 稳定后决定 Capacitor 推送、相机和商店分发范围。

每一阶段完成后必须保持可运行、可测试、可回退，不进行大爆炸式重写。

## 测试与验收

### 自动测试

- CarePlan 调度与 CareOccurrence 生成，包括跨日和时区边界。
- v1→v2 迁移的数据保留、幂等和失败回退。
- 当前宠物切换后五个页面同步更新。
- 快速记录正确分发到各独立领域表单。
- 照护乐观更新成功、失败回滚和撤销。
- 健康、消费、体重、回忆和宠物的现有新增、查看、编辑或删除流程无回归。
- 3D 模块未进入首屏同步包，降级模式可用。

### 视觉与交互验收

- 主要视口为 390×844，同时验证 1440×1024 响应式扩展。
- 无横向溢出、文字截断、底部导航遮挡或安全区问题。
- 首页只突出当前宠物，不平均排列多宠卡。
- 普通页面不出现后台表格化或过量彩色标签。
- reduced-motion 模式关闭视差和非必要动画。

### 完成标准

- 五个一级页面均可从移动导航到达。
- 所有既有功能保持可用。
- 今日照护拥有真实持久化模型和完成历史。
- PWA 可安装且离线可打开应用壳与最近数据。
- `npm test` 和 `npm run build` 通过。

## 主要风险与控制

- **范围过大**：按垂直阶段交付，禁止同时重写五个页面。
- **五项导航拥挤**：标签保持两字、图标统一，并在 390px 真机宽度验收。
- **照护调度复杂**：独立调度模块并覆盖时区/夏令时测试。
- **图片与 3D 性能**：缩略图、懒加载、代码分包和 2D 降级。
- **数据迁移风险**：保留 v1、迁移幂等、验证后写入 v2。
- **离线范围膨胀**：首版只做可安装、离线壳和最近读取，不做同步队列。
